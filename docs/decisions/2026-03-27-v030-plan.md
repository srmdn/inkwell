# Decision Record: v0.3.0 Admin Dashboard

- Date: 2026-03-27
- Status: Accepted
- Owners: @srmdn

## Context

v0.1.0 and v0.2.0 shipped a complete REST API (post CRUD, auth, rebuild, newsletter) and a
reader-facing Astro theme. There is no way to manage content without calling the API directly.
This is the biggest gap between "technically functional" and "actually usable" for someone
installing FolioCMS on their own server.

v0.3.0 closes that gap with a built-in admin dashboard.

## Goals

1. Give users a UI to create, edit, publish, and delete posts without touching the API directly.
2. Keep the admin as part of the core binary, not the theme. Themes handle the reader-facing
   frontend only. Admin is a CMS concern.
3. Unlock `demo.foliocms.com` after this ships. The demo site needs a working admin to be
   worth showing.

## Decisions

### 1. Admin UI is embedded in the Go binary

The dashboard is served by the Go backend as embedded static files. No separate service,
no separate repo, no separate build step for the user.

**Route prefix:** `/admin` (served by the backend, not the theme)

**Rationale:** Self-hosted tools (Ghost, Gitea, Forgejo) embed their admin UI in the main
binary. It keeps the install story simple. Theme authors do not need to build or ship admin
interfaces.

### 2. Stack: React + Vite, bundled and embedded

The dashboard is a React SPA built with Vite. The build output is embedded in the Go binary
via `go:embed`. No CDN, no external scripts at runtime.

**Rationale:** React allows component reuse across editor, post list, and settings. Vite
produces a small, tree-shaken bundle. The Milkdown editor (see decision 4) has a React
adapter, which makes React the natural fit.

**Build step:** `cd admin-ui && npm run build` produces `admin-ui/dist/`. The Go binary
embeds `admin-ui/dist/` at compile time.

**Not used:** Astro (overkill for a SPA), plain vanilla JS (would require reimplementing
state management by hand for the editor interactions).

### 3. Dashboard routes

All admin routes are prefixed `/admin`. The backend serves the embedded SPA for any
`/admin/*` path that is not an API route.

| Path | Description |
|------|-------------|
| `/admin` | Redirect to `/admin/posts` |
| `/admin/login` | Login page |
| `/admin/posts` | Post list (all posts, drafts and published) |
| `/admin/posts/new` | Create post |
| `/admin/posts/:slug/edit` | Edit post |
| `/admin/subscribers` | Newsletter subscriber list |
| `/admin/settings` | Site rebuild trigger, rebuild status |

Auth state is stored in an HttpOnly cookie (same JWT the API already issues). The SPA
reads session state by calling `GET /api/csrf-token` (returns 401 if unauthenticated).

### 4. Markdown editor: Milkdown

The post editor uses [Milkdown](https://milkdown.dev) (v7.x), a ProseMirror-based
WYSIWYG Markdown editor with a React adapter.

**Features used:**
- `@milkdown/preset-commonmark` and `@milkdown/preset-gfm` for full Markdown support
- `@milkdown/plugin-history` for undo/redo
- `@milkdown/plugin-listener` to sync editor state to React
- `@milkdown/plugin-slash` for slash command menu
- Custom toolbar (bold, italic, strikethrough, inline code, headings, blockquote,
  bullet list, ordered list, code block, link, table, horizontal rule)

**Theme:** Custom CSS matching the dashboard dark theme. Not using a Milkdown preset theme.

**Rationale:** Milkdown is open source (MIT), well-maintained, ProseMirror-based (stable
foundation), and has a solid React adapter. It outputs standard Markdown, which is what
the storage layer already reads and writes.

### 5. Image handling in editor: base64 inline (v0.3.0)

For v0.3.0, images pasted or dropped into the editor are converted to base64 data URIs
and stored inline in the Markdown body. No separate upload endpoint.

Hero image (post cover) is a separate file upload field. The file is sent to the save
payload as a base64 data URI and stored as `content/blog/:slug/hero.webp` (converted to
WebP on the backend).

**Rationale:** Keeps the scope manageable. A full media library with S3 support is v0.4.0.
Base64 inline is a known limitation; it is documented as such.

### 6. Media uploads: deferred to v0.4.0

A full media management system (upload, browse, insert) is out of scope for v0.3.0.
The plan for v0.4.0 is a pluggable storage driver interface:

- `LocalDriver`: stores files under `content/media/`, served by the backend
- `S3Driver`: uploads to an S3-compatible bucket (AWS S3, Cloudflare R2, MinIO)

Driver is selected via `MEDIA_STORAGE=local|s3` in `.env`.

### 7. New directory: `admin-ui/`

The dashboard source lives in a new top-level directory in the core repo:

```
admin-ui/
  src/
    components/
      editor/       -- Milkdown editor + toolbar
      posts/        -- post list, post form
      subscribers/  -- subscriber list
      settings/     -- rebuild trigger, status
    pages/
      Login.tsx
      Posts.tsx
      PostEditor.tsx
      Subscribers.tsx
      Settings.tsx
    lib/
      api.ts        -- typed API client (wraps fetch, handles CSRF)
      auth.ts       -- session check helpers
    App.tsx
    main.tsx
  index.html
  vite.config.ts
  package.json
```

The Go backend adds a new route group:

```go
// Serve embedded admin SPA for all /admin/* paths
r.Handle("/admin/*", adminHandler)
```

### 8. No demo site until dashboard ships

`demo.foliocms.com` will be set up as a separate clean install after v0.3.0 ships.
`staging.foliocms.com` is the development instance used to build and test this release.
`foliocms.com` stays as the static landing page only.

## Out of Scope for v0.3.0

- Media library / file browser
- S3 or remote storage (see decision 6)
- Multi-user / roles
- Settings page beyond rebuild trigger (site name, description edits)
- Custom theme management from the dashboard

## Consequences

- A new `admin-ui/` directory and `npm` build step is added to the repo.
  The `install.sh` script will need to run `npm install && npm run build` inside `admin-ui/`
  during setup.
- The Go binary grows slightly due to embedded assets.
- Future contributors need Node.js to build the admin UI (already required for the theme).
